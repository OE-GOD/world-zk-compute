# Glossary

Terminology reference for the World ZK Compute project.

## Proof Systems and Protocols

**GKR (Goldwasser-Kalai-Rothblum)**
An interactive proof protocol for layered arithmetic circuits. The verifier checks the output of a circuit layer-by-layer using the sumcheck protocol, achieving logarithmic verification time relative to circuit size.

**Hyrax**
A polynomial commitment scheme based on Pedersen commitments. Used in this project to commit to the input layer of GKR circuits, enabling the verifier to check that the prover used the correct inputs.

**Groth16**
A succinct non-interactive argument of knowledge (SNARK) with constant-size proofs and constant-time verification. Used as a wrapper around GKR proofs to reduce on-chain verification cost.

**Sumcheck Protocol**
An interactive proof protocol where the prover convinces the verifier of the value of a multivariate polynomial sum. Core building block of the GKR protocol, used at each circuit layer.

**Fiat-Shamir Transform**
A technique for converting interactive proofs into non-interactive ones by replacing verifier challenges with hash outputs. This project uses Poseidon hashing for Fiat-Shamir.

**Poseidon**
An algebraic hash function optimized for use inside arithmetic circuits and zero-knowledge proofs. Used in this project for the Fiat-Shamir transcript (width=3, rate=2, BN254 field).

**MLE (Multilinear Extension)**
A unique multilinear polynomial that agrees with a given function on all Boolean inputs. Used throughout GKR to represent layer values and enable efficient point evaluations.

## Circuit Terminology

**DAG Circuit**
A directed acyclic graph of computation layers, generalizing the linear-layer model. The XGBoost inference circuit uses an 88-layer DAG topology with non-linear atom routing.

**Compute Layer**
A layer in the GKR circuit that performs arithmetic operations (add, multiply) on values from previous layers. Verified via sumcheck during the GKR protocol.

**Input Layer**
The bottom layer of a GKR circuit containing the actual input values. Verified via Hyrax (Pedersen) commitment openings.

**Atom**
A single gate or operation within a compute layer. Each atom references operands from source layers and applies a product-of-operands expression.

## Zero-Knowledge Virtual Machines

**zkVM (Zero-Knowledge Virtual Machine)**
A virtual machine that executes programs and produces a cryptographic proof that the execution was correct. Examples: RISC Zero, SP1.

**Guest Program**
The program that runs inside a zkVM. It reads private inputs, performs computation, and commits public outputs. Compiled to RISC-V (RISC Zero) or RISC-V (SP1).

**Host**
The process that runs outside the zkVM. It provides inputs to the guest, invokes the prover, and receives the proof and public outputs.

**Journal**
The public output committed by a RISC Zero guest program. The journal is included in the receipt and is visible to anyone verifying the proof.

**Seal**
The cryptographic proof data in a RISC Zero receipt. Verifying the seal confirms the journal was produced by a specific guest program (identified by image ID).

**Receipt**
A RISC Zero proof artifact containing the journal (public output) and the seal (cryptographic proof). Can be verified on-chain or off-chain.

**Image ID**
A cryptographic hash that uniquely identifies a compiled guest program binary. Used to verify that a proof was generated by a specific program version.

## TEE (Trusted Execution Environment)

**TEE (Trusted Execution Environment)**
Hardware-isolated execution environment that protects code and data from the host OS. Used as a fast path for inference with ZK as a dispute fallback.

**Enclave**
A specific TEE instance (e.g., AWS Nitro Enclave). Runs the inference workload in isolation and produces a signed attestation of the result.

**Attestation**
A cryptographic document produced by TEE hardware that certifies what code is running inside the enclave and what it produced. Includes measurements (PCR values) signed by the hardware.

**PCR (Platform Configuration Register)**
A hash measurement of the enclave image and configuration. PCR0 corresponds to the enclave image hash. Used to verify the enclave is running the expected software.

## Ethereum and Smart Contracts

**EIP-170**
Ethereum standard limiting deployed contract bytecode to 24,576 bytes. Relevant for L1 deployments; L2s like Arbitrum allow larger contracts.

**EIP-1967 (UUPS Proxy)**
A proxy pattern where the upgrade logic lives in the implementation contract. Storage slots for the implementation address and admin address are defined at fixed, non-colliding locations.

**Prover**
An entity that generates cryptographic proofs. In the operator model, provers claim execution requests, run the computation, and submit proofs on-chain.

**Verifier**
A smart contract that checks cryptographic proofs on-chain. Different verifiers handle different proof types (RISC Zero receipts, GKR proofs, Groth16 proofs).

**Requester**
An entity that submits an execution request on-chain, specifying the program (image ID), input, and payment. Provers compete to fulfill requests.

## Dispute and Finalization

**Challenge Window**
A time period after a TEE result is submitted during which anyone can dispute the result by posting a bond and requesting a ZK proof.

**Dispute**
A challenge to a TEE-submitted result. The challenger posts a bond, and a ZK proof is generated to determine correctness. The losing party forfeits their bond.

**Finalization**
The point at which a result becomes accepted and immutable. For TEE results, this happens after the challenge window closes without dispute. For ZK results, this is immediate upon proof verification.

## Deployment

**Registry (deployments/registry.json)**
An off-chain JSON file tracking deployed contract addresses, versions, and transaction hashes per chain.

**Chains (deployments/chains.json)**
Configuration file listing supported chains with RPC URLs, chain IDs, gas limits, and deployment notes.
